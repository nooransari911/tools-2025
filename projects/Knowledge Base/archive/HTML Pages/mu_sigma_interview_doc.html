—> Backup Project

The project is an automated data backup pipeline in AWS. The core objective is a powerful backup solution that has incremental nature, integration with git and high degree of automation.

The pipeline uses a two-stage architecture to ensure robust and durable backups. First, an initial rsync operation consolidates all local content into a local staging directory. Then, another rsync operation updates from staging area to S3

In addition to the core backup function, the project also generates a dataset. The dataset contains fine-grained logs of all S3 operations. I developed a suite of SQL queries to analyze these logs and get real-time S3 usage insights.

Tell me about a challenge/achievement

I was building a data pipeline that required two different libraries. They worked fine in isolation. But together, they were incompatible. The whole program would crash instantly.

I looked into the error. It turned out both libraries were fighting over a shared, low-level component. They were trying to register the same definition, which caused the conflict.

We couldn’t edit the libraries directly. So, I wrote a small patch to fix it. It would let the first library load the component normally. When the second library tried to load the component, my patch would intercept the request and return the component that was already loaded.

This simple fix lets both libraries run together.

Personal Finance Visualization App

Why did you choose BigQuery for this project?

My choice of BigQuery was driven by the primary technical requirement: a serverless, pay-per-query SQL database.

A provisioned database like RDS is very powerful and standard. But it would be idle 99% of the time and be extremely uneconomical. BigQuery was a natural fit for me as it charges only for queries I run and amount of data stored.

What was the most challenging part?

The most interesting design challenge was handling SQL query generation. Query would need to be constructed based on user inputs. I immediately ruled out hardcoded queries corresponding to specific months/years as they would be a nightmare to maintain.

Personally, maintainability was paramount in spite of the relatively simple nature of the project. So I created a query generation engine. It is a parameterized function that constructs SQL WHERE clauses based on user inputs. While the final implementation is quite straightforward, it needed quite a few iterations to get right

What insights did this project reveal?

The main objective of this project was not to reveal new insights. I’m already quite financially disciplined and the volume of data is quite small/ Instead, the project’s main success has been transformation of raw unintelligible transaction data into a queryable asset. The project has been invaluable in validating and grounding my own understanding of my spending habits.

Tell me about the data. What specific cleaning or transformation steps did you perform?

The raw data came from the bank’s website in Excel format.

When I started working with the files, I quickly realized that specific column ordering was not guaranteed. My pipeline would need to be resilient against changes in column ordering. So I implemented dynamic header parsing to dynamically discover essential columns like amount, date, etc.

I also performed data normalization on the ‘Payee’ column. Many entries had generic prefixes such as UPI-. To improve subsequent queries and readability, I used regex to parse these strings and extract the core payee name. Then the payee name was limited to 20 characters to ensure readability. This transformed the raw, messy entries into a clean and consistent format, which is critical for meaningful aggregation and analysis.

Why build a custom Flask app instead of using a standard BI tool like Looker Studio?

That’s a great question that gets to the heart of motivation for this project. While BI tools are an excellent option for rapid analysis, they would not meet my 2 critical imperatives.

First, I needed to maintain portability and minimize vendor lock-in. My architecture uses standard Python libraries and SQL. This ensures clean separation of concerns. If required, BQ could easily be swapped out for another SQL-compliant database. Off-the-shelf BI tools lack this degree of portability.

Second, since this was sensitive personal financial data, maintaining privacy was paramount. BI tools would not provide absolute privacy or control. A custom self-hosted Flask app ensured maximum privacy and control.

How would your pipeline change if you needed to process statements from 10 different banks, each with a different CSV format?

The system was designed with that kind of extensibility in mind. My current implementation uses dynamic header parsing. It is already resilient against variations in column orders. To handle more complex variations, we would need to implement a dedicated parser module.

The module would map all column names in different files to standardized internal column names. This allows seamless ingestion of multiple bank statements and ensures maintainability.

AI Document Processing in Python

Can you give a concrete use case?

Absolutely. The project was invaluable when I needed to synthesize study notes for a course under tight time deadlines. The material for the course was spread across more than 50 files. The files were organized logically by modules/units.

The file organization meant that simple serial/parallel processing would not be sufficient. So I decided to extend the platform to work with the specific file organization. I quickly developed and integrated a new module that could work with the file organization.

This extension allowed me to process the entire course in under 10 minutes. The task done manually would have taken at least a day.

The real success story here was not only was I able to save a day’s worth of time, but also the platform was able to prove its extensibility and could effortlessly adapt to new workflows and requirements. That, in my opinion, was the real win.

How did you implement the flexible schema for structured output?

To enforce clean separation of concerns, I isolated schema implementation from the core platform. Schemas are stored in a separate repository. The platform simply dynamically sources specific schemas at runtime.

This implementation ensures that the core platform is strictly isolated from usage of various schemas or changes in schemas.

How did you measure the quality or accuracy of the AI-generated output?

For structured data output, the platform would perform validation using Pydantic. Output is written to the primary output files only if the output passes validation. Otherwise it is quarantined into separate files for review. This creates a strong guarantee that any data written to the primary output files have valid schemas.

For other content types, I used an iterative feedback loop. I would process a small batch of documents, manually assess the outputs, and then refine the prompts based on my assessments. This proved to be a highly effective way to ensure the quality of generated outputs

What happens if an external API like Claude is down or returns a rate-limit error?

All external API calls are wrapped in robust try-except blocks.

Additionally, code returns null upon API exceptions. Critically, the subsequent downstream logic is designed to handle these nulls. This makes the platform resilient against API-/network-related exceptions.

Absolutely. The project’s clean separation of concerns was invaluable in making the platform cost-effective.

After the platform was initially developed, it had support for flash and pro gemini models. However, I quickly realized that flash and pro formed a dichotomy in cost and performance. Therefore, I began exploring other options.

Automated Backup Solution with AWS S3

Why did you use a two-stage architecture for the backup?

The two-stage architecture was a deliberate choice to ensure data consistency and integrity. I needed to back up multiple independent local directories. Backup triggered automatically could result in inconsistent states, or worse, propagate user errors or corrupted data to backup as well.

The first stage syncs all local content into a single local staging directory. This creates a consistent, temporal snapshot. The second stage then syncs this stable, unified snapshot to S3. This provides a simple way to ensure data consistency and integrity.

What insights did you get from analyzing the S3 operation logs?

My initial plan was to build an analytics pipeline by querying the raw CloudTrail logs. CloudTrail logs are stored in S3, partitioned appropriately to make them suitable to be queried by Athena and finally, analyzed by SQL queries in Athena.

However, this had multiple critical issues: the CloudTrail data itself was incredibly noisy; Athena required arcane prefix patterns for partitioning; and Athena’s proprietary SQL dialect had its own quirks.

So I settled on using CloudWatch dashboards. It gave me all the insights I required and was very user-friendly.

The key takeaway for me was that a ‘clever’ architecture is sometimes backwards (“a solution looking for a problem”). I learned the immense value of choosing boring, reliable tools. I should have chosen a proper database for logging instead of object storage masquerading as a database S3.

How do you monitor this pipeline for silent failures? How would you know if it hadn't run successfully?

I made a deliberate design choice to use a manually-triggered, human-in-the-loop process over fully automated, scheduled automation. Backup triggered automatically could result in inconsistent states, or worse, propagate user errors or corrupted data to backup as well. This ensures absolute data integrity and consistency over convenience.

There is no unattended run that could fail silently or catastrophically. As an additional fail-safe, the program automatically runs dry runs before executing actual operations. The dry runs gives a streamlined preview of affected files in backup. This helps minimize user error and ensures I have full visibility of files being updated in backup.

How did you manage credentials and ensure the security of the data?

I followed standard AWS security best practices. For credential management, access credentials are not hardcoded. Instead, the program leverages AWS CLI’s secure credential management system, which automatically and securely sources credentials from local configuration files.

AWS CLI uses authenticated HTTPS connections for all API calls. Furthermore, my destination S3 bucket uses SSE-S3, which is server-side encryption-at-rest for data in S3. This provides comprehensive security for both data at rest and data in transit.
